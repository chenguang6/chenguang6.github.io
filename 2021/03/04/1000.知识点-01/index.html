排序算法

冒泡排序：

  • 每次拿两个数进行比较，总是把小的放在最前面，大的放在最后面，第二次是第二个数和第三个数比较，如果第三个小于第二个，那么第三个和第二个替换替换，依次这样比较替换，第一轮循环完了，放在最后面的那个数就是这一串数字中最大的数。第二轮循环和第一轮一样，第一个和第二个比较，最小的放在最前面，依次类推，如果有10个数，那么就循环9次，最小的在最前面，最大的在最后面。

    二分查找：

  • 原理就是必须有一串从小到大的排序，把这串数字进行划分，分为三段，前半段，中止段，和中止后半段。查找数字或者字符先从中止段查找，查到的数字或者字符和中止真好相等，那么就直接取出来，如果比它大就从后半段查找，后半段在进行二分法，进行递归的调用查找，如果比中止小，走前面的中止段，在进行拆分，进行二分查找法。

Spring

  • Spring是一个主流的Java Web 开发框架，该框架是一个轻量级的应用框架。

  • Spring它的核心思想有两个，IOC（控制反转）、AOP（依赖注入）

1）方便解耦，优化开发

  • Spring就是一个大工厂，可以将所有对象的创建和依赖关系的维护交给Spring管理

2）方便集成各种优秀的框架

  • Spring不排斥这种优秀的开源框架，内部提供了对各种优秀框架的直接支持（如 Struts2、Hibernate、MyBatis 等）

3） 降低 Java EE API 的使用难度

  • Spring对Java EE开发中非常难用的一些API都提供了封装，使用这些API应用的难度大大降低（JDBC、远程调用等）

4）方便程序的测试**

  • Spring支持Junit4，可以通过直接方便的测试Spring程序

5）AOP编程的支持**

  • Spring提供面向切面编程，可以方便地对程序进行权限拦截和运行监控等功能

6） 声明式事务的支持**

  • 只需要通过配置就可以完成对事务的管理，无需手动编程

依赖注入：在之前的开发，如果想创建一个对象，就new一个，如果想这个对象中定义其他的变量或者对象，就在对象内部创建一个成员变量，但是在spring中，咱么可以在配置文件中配置一个bean，指定对应的全路径名称。如果想用这个类的对象，在代码中读取这个配置文件，然后就可以直接获取到这个类的对象，当前使用注解的话，更方便，如果想在某个对象中注入某些属性，可以给一个property的属性，这个就是注入

控制反转：对于控制翻转，和DI是一个道理，也是指我们不需要new对象，可以直接从配置文件中将对象取出，通过反射的方式

AOP：面向切面编程，把传统程序执行过程看成纵向执行过程，每个方法当做一个点，基于这些点可以进行增强处理，形成了纵向的切面，包含了原有方法和增强方法，不改变原有代码结构，增添额外功能，常用于事务管理、日志发音等，其实Spring的AOP就是对代理设计模式的一种实现

分步式

  • 现在大部分公司都使用的分布式架构，我们根据模块来把项目拆分成一个个的工程，这样我们各个模块之前的耦合度就降低了，然后我们各个模块之前通过接口来相互通讯，我们公司用的是dobbo来实现项目之前的通讯，这个我们的代码拓展性就更好了，最后我们在部署项目的时候，可以根据用户的访问量来给各个模块的服务器增大配置，我们还可以采用nginx负载均衡实现一个反向代理，当用户请求后台服务器的时候，由nginx来决定那台服务空闲或者压力小，来让用户访问，防止高并发

单例模式如何实现的

其实单利的本质就是控制对象的产出，要想产生一个唯一的实例，你就要私有化构造方法，然后私有化一个静态的全局变量，并提供一个公共的静态方法供外界访问，单利大概就是这样

单例的实现方式：

饿汉式（**直接创建对象，不存在线程安全问题，在类初始化时直接进行实例化****）

直接实例化饿汉式（简介直观）
枚举是（最简洁）
静态代码块饿汉式（适合复杂实例化）

懒汉式（延迟创建对象）

线程不安全（适用于单线程）
线程安全（适用于多线程），双重检查加锁
静态内部类形式（适用于多线程）

总结：

  • 如果是饿汉式，枚举形式最简单
  • 如果是懒汉式，静态内部类形式最简单

单例的运用场景：

  • 项目配置文件：读取项目的配置信息的类可以做成单例的，因为只需要读取一次，且配置信息字段一般比较多节省资源。通过这个单例的类，可以对应用程序中的类进行全局访问。无需多次对配置文件进行多次读取。
  • 应用日志类：日志器Logger在你的应用中是无处不在的。也应该只初始化一次，但是可以到处使用。

# 和 $ 号的区别

  • 这两个符号一般是在使用MyBatis编写底层SQL语句时使用，#就是一个占位符，具体的使用是#{id},而$是一个原样输出的标识，是${value}，我们再项目用#，这样可以防止SQL注入，安全性高

hashmap的底层实现原理

  • hash底层是通过数组和链接联合实现的，当我们创建hashmap时会先创建一个数据，当我们用put方法存数据时，先根据key的hashcode值计算出hash值，然后用这个哈希值确定在数据中的位置，再把alue值放进去，如果这个位置本来没有放东西，就直接放进去，如果之前就有，就会生成一个链表，把新放入的值放在头部，当用get方法获取值时，先根据key的hashcode值计算出hash值，确定位置，再根据equals方法从该位置上的链表中取出该value值

简单说一下SpringMVC和ssm的整合

  • 简单的说SpringMVC在ssm中整合，就是在web.xml里面配置springMVC的核心控制器：DispatcherServlet；他就是对指定的后缀进行拦截；然后再springMVC.xml里边配置扫描器，可以扫描带@controller注解的这些类，现在用springMVC都是基于注解开发的，像@service、@Repository，@RequestMapping和@Responsebody等这些注解标签，都是开发时用的，每个注解标签都有自己的作用，他还配置了一个视图解析器，主要就是处理之后的跳转进行统一配置

Mybatis和Hibernate的区别

  • Hibernate一个全封装，mybatis是半封装，使用hibernate做表单查询操作的时候比较简单（因为hibernate是针对对象进行操作的），但是夺标查询起来就比较繁琐了，比如说5张表10张表做关联查询，就算是SQLQuery那后续的维护工作也比较复杂，还有就是Hibernate再Sql优化上执行效率上会远低于MyBatis（因为Hibernate会把表中所有的字段查询出来，比较消耗性能）我们以前在做传统项目方面用hibernate，但是现在基本行都在使用mybatis

Dubbo

什么 是Dubbo？

  • Dubbo 是一个分布式服务框架，使用rpc（远程过程调用协议）协议进行远程调用，直接使用socket通信。因为是通过二进制进行传输的，所以传输效率高，并且可以通过检测中心统计出系统之间的调用关系，调用次数。

为什么使用Dubbo

  • 当网站变大后我们需要拆分应用进行服务化，来提高开发效率，或者是调用网站性能，当然现在我们写项目都是分布式项目，及时是做后台管理，现在公司里好多也是使用SOA架构模式，也就是服务层和表现层分离，分离就产生一个问题，怎样提供接口，怎样调用Dubbo可以发布服务，也给我们提供服务的方式，现在好多公司都用这种方式来做了。

Dubbo执行过程

  • 我先来说一下这个Dubbo的调用过程吧，dubbo里面几个比较关键的点，一是服务提供者，就是提供服务的一方，还有一个消费者，用来消费所提供的的服务接口，还有一个注册中心，就是服务注册到那里，我们用的是zookeeper，一般是配置到linux服务商，还有检测器，能检测服务发布和调用，还有一个服务容器，就是tomcat或者其他服务器。
  • 它的执行过程，我们写项目时，就会在pom文件中配置duvvo和zookeeper，然后会在spring的配置文件中配置要发布的服务接口，以及服务要发布到注册中心的ip及端口。这样当程序跑起来的时候，dubbo这个框架爱就类似于spring，它会读取配置文件中配置信息，把注册的服务，注册到zookeeper这个注册中心去。
  • 我们写表现层时，就在配置文件中配置好要调用的服务接口，和zookeeper中心所在服务器的ip端口，程序一启动，依然是dubbo这个框架读取配置文件中服务的信息，根据接口名称就知道要调用哪个服务。
  • 服务层项目会不断的向注册中心发布自己的服务，表现层项目也会不断的到注册中心去查找服务，要是配置的接口不对，或者zookeeper这个注册中心没有启动，那这个项目就没法玩了。

Dubbo原理

关于dubbo的原理，我也是稍微研究了一下，可能不是太透，dobbo的框架设计的话一共分为10层，我就说一下我记住的那些吧。

最外层的哈，是接口或者服务层，就是我们配置文件中所暴露的接口 ，这个是我们能看到的，然后再往下，是配置层，主要负责读取我们的配置文件，把这些配置信息读取出来，还有一个服务代理层，就是负责代理我们所写的服务执行对应功能，就像Spring中JDK代理类似。再往下，是服务注册层，读取完配置文件信息，需要将dubbo的配置信息注册到注册中心，再走的话，有一个监控层，可以检测rpc协议执行过程中的调用测试及调用时间。再有信息交互、网络传输层，这里才是进行网络的传输，进行的是socket通信，还有一个数据序列化，这一层就是要传输的数据对象进行序列化，所以我们在写代码时需要让对象实现序列化接口（可以避开原理这块，聊不好会死人的！！）
